roborts−baseroborts-baseroborts−base 源码解析

1.简介

本功能包用以进行ros节点和下位机的交互工作，在自动化框架中充当着“信息中转站"的角色通过ros通信接收各功能包发布的信息，再将这些信息,通过虚拟串口发布给下位机。

2.虚拟串口

  • 上位机收发基础知识
    Linux将外部设备当做文件，所以简单的虚拟串口应用便不难理解。使用USB线将Linux与下位机相连后，只需要将数据处理成下位机串口收发需要的形式，然后打开文件，写入数据即可完成通信。（详细教程请点击这里）。
  • 本功能包实现
    • 上位机向单片机发布示例：
      chassis_speed_pub_ = handle_->CreatePublisher<roborts_sdk::cmd_chassis_speed>
(CHASSIS_CMD_SET, CMD_SET_CHASSIS_SPEED,MANIFOLD2_ADDRESS, CHASSIS_ADDRESS);
      建立发布者时模板cmd_chassis_speed为消息类型，第一个参数为消息分区，从0x00u到0x04u, 具体定义为：
      //CMD_SET                            
#define UNIVERSAL_CMD_SET              (0x00u)
#define REFEREE_SEND_CMD_SET           (0x01u)
#define CHASSIS_CMD_SET                (0x02u)
#define GIMBAL_CMD_SET                 (0x03u)
#define COMPATIBLE_CMD_SET             (0x04u)
      可以看到，上文中第一个参数CHASSIS_CMD_SET定义为0x02u
      第二个参数为消息类型，在不同的分区中从0x01u开始依次排列
      第三个参数为发布者，第四个为接收者。这两个参数使用预先定义好的底盘、云台、上位机，同样从0x00u开始，这样发布消息就做到不乱，也不漏了
    • 上位机接收单片机;
      与发布完全相同，只是第三个和第四个参数换了个位置

3.文件目录

  • roborts_base_node: 启动节点，逐句解析如下
    int main(int argc, char **argv){
ros::init(argc, argv, "roborts_base_node");//初始化ros节点“roborts_base_node”
ros::NodeHandle nh;                        //创建节点句柄
roborts_base::Config config;               //这个类明确了硬件设备的名字
config.GetParam(&nh);
//创建一个handle类，主要协议内容见后文
auto handle = std::make_shared<roborts_sdk::Handle>(config.serial_port);

if(!handle->Init()) return 1;              //协议句柄初始化

roborts_base::Chassis chassis(handle);     //创建底盘类（会调用构造函数）
roborts_base::Gimbal gimbal(handle);       //创建云台类（会调用构造函数）
roborts_base::RefereeSystem referee_system(handle); //一个很杂的类，
while(ros::ok()) {                                  //用于视觉和裁判系统

    handle->Spin();    //在上文类的构造函数中，创建了接受者对象，接收者对象有回调函数
    ros::spinOnce();   //从而可以不断地被这两个函数回调，
    usleep(1000);//休眠1000微秒
}
}
  • chassis :底盘控制节点（云台控制可以类比）
    • 简介
      该节点定义了一个类chassischassischassis，这个类的构造函数会调用ROS_Init()和SDK_Iinit()两个初始化函数，前者创建了订阅者对象（好像是静态的定义吧具体没研究），并在订阅者的回调函数中按需调用了ros发布者和虚拟串口发布者，从而完成了消息传递的闭环。
    • 收发消息汇总（消息类型可在roborts_msgs包中查看）
      • 下位机传入后由ROS发布（todo：下位机发布协议）
        1. 话题 “odom”：包括一个即时时间戳，以及该时刻的xy线速度、起点坐标、朝向和角速度（可能来自陀螺仪角度和转速？）
        2. 坐标系变换 “odom_tf”: 发布了一个以“odom”为父坐标系，“base_link”为子坐标系的坐标变换，变换值是当前位置（里程计：在运动开始时，里程计坐标系“odom”与世界坐标系“map”是重合的，理想状态下，这两个坐标系是不动的，base_link为机器人自身坐标，随机器人移动而变化，当机器人出现飘移，里程计坐标系和世界坐标系才会分开）
        3. 话题 “uwb”： UWB基站定位用的，应该已经不用了，不过定位包中这部分代码仍然存在（todo）
      • ROS传入后向下位机发布
        1. 话题 "cmd_vel"：发布x、y方向线速度，定义了一个vw存储角速度，rotate_x、rotate_y置零（迷惑操作）
        2. 话题 “cmd_vel_acc”: 发布x、y方向线速度和加速度，定义了一个vw存储角速度和角加速度，rotate_x_offset、rotate_y_offset置零
  • gimbal :云台控制节点
    • 简介（和底盘雷同）
    • 收发消息汇总
      • ROS节点传入后向下位机发布
        1. 话题 "cmd_gimbal_angle"：发布了云台的状态，以及pitch和yaw的角度（视觉包发布）
      • 下位机传入后ROS发布
        1. 坐标系变换 "gimbal_tf_": 发布父坐标系"base_link",子坐标系"gimble"之间的坐标系变换，偏向角为下位机传入的pitch和yaw数据，坐标系为写死的0 0 0.15（todo：下位机发布协议）
      • 服务
        1. 服务 "set_gimbal_mode": 把从下位机传入一个uint8，作为req，收到请求后发布给下位机，res赋值为ture，是云台模式的选择(官方包未发现客户端)
        2. 服务 "cmd_fric_wheel"：收到请求后，发布摩擦轮速度（两档），res=true （官方包未发现客户端）
        3. 服务 "cmd_shoot"：请求包括是是否射击以及射击的弹丸数or持续射击，res=true（官方包未发现客户端）
